# AE000:054 spectrogram diagnosis

The user compared the real in-game capture against the Audio Atlas `adlib_like` WAV. The lower/atlas render has too many clean high-frequency horizontal bands and sounds brighter/thinner than the game.

Important conclusion: this does **not** mean the DAT/EXE FM instrument table is wrong. The project's Python `adlib_like` renderer is still only a simplified preview synth. It uses the recovered OPL patch table and voice layout, but it is not a YM3812 emulator.

Findings from the ASM and spectrum:

* `AE000:054` really uses the AdLib/Sound Blaster OPL path: the header supplies 9 instrument ids and `D8F0 -> DA66 -> E0C0/E1F2/E27B/E2D6/E324/E372/E44B` programs OPL registers.
* The note layer is `stream 0 -> voices 0..2`, `stream 1 -> voices 3..5`, `stream 2 -> voices 6..8`; `voice cfg` bytes are added to the note index before `E48A` writes `A0/B0`.
* `E48A` does not convert arbitrary Hz to FNUM. It indexes runtime tables around `C6C3/C5EA/C64A/CA24/CA6D`, generated by `DEFA/DE7E/DDD9`. The current VGM/trace still approximates this with the YM3812 formula and a global transpose.
* The previous Python preview made the timbre too bright because it applied only the carrier envelope. In real FM, the modulator has its own envelope; when the modulator decays, the high sidebands decay too. The preview now applies the modulator envelope as the FM-index envelope.
* The previous Python preview used a gentler total-level curve. The preview now uses the YM3812 total-level scale of 0.75 dB/step.

Remaining accuracy blocker:

The atlas WAV should not be treated as authoritative. For faithful output we should render the exported VGM/register trace through a real OPL core such as Nuked OPL, DOSBox/DBOPL, libADLMIDI, or another YM3812 emulator. If the VGM still differs after real OPL rendering, then the next target is exact reconstruction of `E48A`'s runtime FNUM/block tables and the `CA6D/CA24` pitch table selection path.
